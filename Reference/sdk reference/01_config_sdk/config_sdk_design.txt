====================================================================
TECHNICAL DESIGN SPECIFICATION
CONFIG_SDK
AGENTIC AI MDLC / ECL PLATFORM
VALIDATION-GRADE, HUMAN-TRACEABLE, FUTURE-PROOF
====================================================================

STATUS
--------------------------------------------------------------------
Draft v3.0

PURPOSE
--------------------------------------------------------------------
config_sdk is the central configuration and configuration-governance
foundation for the agentic AI MDLC platform.

It is responsible not only for loading, validating, merging, and
resolving configuration, but also for ensuring that every material
configuration state, change proposal, review, challenge, response,
and decision can be:

- understood by humans later
- validated independently from workflow runtime
- exported as evidence
- linked to exact config versions and hashes
- traced across time, actors, and approval stages
- defended in model validation, audit, and governance review

This SDK must therefore support both:
1. machine execution and config resolution
2. human governance and validation traceability

====================================================================
1. DESIGN INTENT
====================================================================

config_sdk must serve as:

1. The single source of truth for effective configuration
2. The config contract layer for all downstream SDKs
3. The config provenance engine
4. The config governance engine
5. The config traceability engine
6. The config evidence assembly layer
7. The config audit snapshot producer

This means config_sdk is no longer only a parser / resolver.
It becomes a governed configuration backbone for the platform.

====================================================================
2. PRIMARY DESIGN OBJECTIVES
====================================================================

OBJECTIVE A: DETERMINISTIC CONFIGURATION
--------------------------------------------------------------------
Provide deterministic, reproducible config resolution across all SDKs.

OBJECTIVE B: EXPLAINABLE CONFIGURATION
--------------------------------------------------------------------
Allow every effective value to be traced back to:
- source layer
- source file
- override chain
- applicable scope
- effective hash

OBJECTIVE C: GOVERNED CONFIGURATION CHANGE
--------------------------------------------------------------------
Allow material config actions to be proposed, reviewed, responded to,
approved, rejected, or superseded in a structured way.

OBJECTIVE D: HUMAN VALIDATION TRACEABILITY
--------------------------------------------------------------------
Allow validators, auditors, and reviewers to reconstruct:
- what changed
- why it changed
- who proposed it
- who challenged it
- who approved it
- under what evidence and assumptions
- for which exact config version/hash

OBJECTIVE E: INDEPENDENT QUERYABILITY
--------------------------------------------------------------------
Reasoning, review, decision, and trace records must be queryable
independently from workflow state.

OBJECTIVE F: EXPORTABLE EVIDENCE
--------------------------------------------------------------------
Support immutable audit snapshots and evidence bundles suitable for:
- model validation packs
- audit trail review
- committee review
- issue investigation
- release approval evidence

OBJECTIVE G: FUTURE-PROOF AGENTIC COMPATIBILITY
--------------------------------------------------------------------
Support agent-driven config recommendations while preserving human
control, challenge, and evidence traceability.

====================================================================
4. WHY THIS VERSION IS VALIDATION-GRADE
====================================================================

A normal config engine answers:
- what is the effective config?

A validation-grade config engine must also answer:
- why is this the config?
- what changed from previous state?
- who proposed the change?
- what assumptions were made?
- what evidence supported it?
- who reviewed or challenged it?
- how was the issue resolved?
- who approved or rejected it?
- what exact config version/hash was in force?
- what was the final traceable state at the time?

This SDK is designed specifically to answer those questions later.

====================================================================
5. ARCHITECTURAL POSITION
====================================================================

config_sdk sits at the bottom of the platform dependency graph.

All downstream SDKs depend on it for config consumption and governed
config evidence.

config_sdk itself should remain minimal in external dependency and
must not depend on:
- workflow_sdk
- registry_sdk
- artifact_sdk
- domain SDKs
- bridge SDKs

To preserve bottom-layer purity, it should expose pluggable repository
interfaces for persistence rather than hard-coding infrastructure
dependencies.

6. KEY DESIGN PRINCIPLES
====================================================================

PRINCIPLE 1: CONFIG IS A GOVERNED ASSET
--------------------------------------------------------------------
Configuration is not just runtime input. It is a governed artefact.

PRINCIPLE 2: EFFECTIVE CONFIG MUST BE REPRODUCIBLE
--------------------------------------------------------------------
The same inputs must produce the same effective config and same hash.

PRINCIPLE 3: EVERY MATERIAL CHANGE MUST BE TRACEABLE
--------------------------------------------------------------------
Material changes must carry reasoning, review, and decision context.

PRINCIPLE 4: HUMAN TRACEABILITY IS A FIRST-CLASS REQUIREMENT
--------------------------------------------------------------------
Data structures must support later human understanding without forcing
reviewers to reconstruct history manually.

PRINCIPLE 5: WORKFLOW STATE IS NOT THE SYSTEM OF RECORD
--------------------------------------------------------------------
Workflow may reference governance, but governance records must remain
independently queryable.

PRINCIPLE 6: CONFIG CHANGE MUST HAVE BEFORE/AFTER SEMANTICS
--------------------------------------------------------------------
A human reviewer must see precisely what changed.

PRINCIPLE 7: ACTORS MUST BE EXPLICIT
--------------------------------------------------------------------
Human, agent, and system roles must be distinguishable.

PRINCIPLE 8: EVIDENCE MUST BE STRUCTURED
--------------------------------------------------------------------
Evidence references must be explicit and linkable.

PRINCIPLE 9: TIMELINE RECONSTRUCTION MUST BE POSSIBLE
--------------------------------------------------------------------
The SDK must support ordered narrative reconstruction of config events.

PRINCIPLE 10: SNAPSHOT EVIDENCE MUST BE EXPORTABLE
--------------------------------------------------------------------
A point-in-time audit package must be producible.

11. GOVERNANCE MODEL
====================================================================

The governance model separates 5 layers:

1. PROPOSAL LAYER
--------------------------------------------------------------------
Captures suggested action and rationale.

2. CHANGE LAYER
--------------------------------------------------------------------
Captures exact before/after change set.

3. REVIEW LAYER
--------------------------------------------------------------------
Captures challenge, response, clarification, conditions, unresolved items.

4. DECISION LAYER
--------------------------------------------------------------------
Captures formal outcome and authority.

5. SNAPSHOT LAYER
--------------------------------------------------------------------
Captures immutable evidence at point in time.

This separation is important because later validation may need to
inspect one layer without replaying all workflow state.

25. GOVERNANCE ORCHESTRATION RULES
====================================================================

The SDK should support configurable governance triggers such as:

- material change requires review
- critical key path requires approval
- runtime override requires reason note
- schema exception requires decision

====================================================================
29. HUMAN VALIDATION USE CASES
====================================================================

USE CASE 1: MODEL VALIDATION CHALLENGE
--------------------------------------------------------------------
Validator asks:
Why was prod review gating enabled for this project?

Expected answer path:
- reasoning note
- change set
- review request
- response
- approval decision
- effective config hash
- provenance of key

USE CASE 2: AUDIT INVESTIGATION
--------------------------------------------------------------------
Audit asks:
Who approved the schema exception and under what basis?

Expected answer path:
- decision record
- basis type
- decided_by_actor
- linked evidence
- linked review
- snapshot pack

USE CASE 3: RELEASE RECONSTRUCTION
--------------------------------------------------------------------
Reviewer asks:
Which config was live at release and what approvals existed?

Expected answer path:
- effective config hash
- decision records effective at release
- release snapshot

USE CASE 4: OVERRIDE DEFENSIBILITY
--------------------------------------------------------------------
Stakeholder asks:
Why did runtime override replace project value?

Expected answer path:
- reasoning
- materiality
- authority level
- review/decision trail

34. ACCEPTANCE CRITERIA
====================================================================

config_sdk is acceptable only if all the following are true:

1. It resolves configuration deterministically.
2. It validates config reliably.
3. It explains origin of effective values.
4. It captures structured reasoning records.
5. It captures explicit change sets with before/after values.
6. It captures structured review requests and responses.
7. It captures formal decision records with authority and basis.
8. It keeps decisions queryable outside workflow state.
9. It supports structured actor identity and role.
10. It supports ordered timeline reconstruction.
11. It can assemble a human-readable trace bundle.
12. It can create immutable audit snapshots.
13. It links governance records to exact config hash/version.
14. It supports later human validation without requiring manual table joins.
15. It remains bottom-layer safe and future-proof.

This enhanced config_sdk is designed to ensure that configuration is:

- deterministic for execution
- explainable for developers
- governable for reviewers
- defensible for validators

DESIGN INTENT
--------------------------------------------------------------------
The public API surface of config_sdk must remain:

- small
- stable
- strongly typed
- deterministic
- easy for downstream SDKs to consume
- rich enough for governed configuration traceability
- backward-compatible in spirit with the initial config_sdk design

The public API must preserve the original core responsibilities:
- loading config
- validating config
- resolving config stacks
- rendering effective config
- explaining provenance
- diffing config
- hashing config
- exporting effective config

It must also extend the surface for:
- reasoning capture
- change set capture
- review lifecycle
- decision recording
- trace bundle generation
- audit snapshot generation
- independent governance querying

This means config_sdk public API is divided into 5 layers:

1. Core resolution API
2. Validation API
3. Governance API
4. Traceability API
5. Snapshot / evidence API

The main user-facing service remains:
- ConfigService

An optional compatibility alias may remain:
- PublicService
